AI 기반 시각장애인 PC 음성 비서 시스템

Voice Navigator

최종 기획서

1. 프로젝트 개요

Voice Navigator는 시각장애인 및 저시력 사용자가 Windows PC와 웹 서비스를 보다 쉽게 사용할 수 있도록 돕는 AI 기반 음성 비서 시스템이다. 본 시스템은 단순히 화면의 텍스트를 낭독하는 스크린리더의 역할에 머무르지 않고, 사용자의 명령을 이해한 뒤 실제 작업을 대신 수행하고 그 결과를 요약하여 안내하는 것을 목표로 한다.

예를 들어 사용자가 '유튜브 들어가서 고양이가 춤추는 영상 찾아줘' 또는 '네이버 지도에서 서울역에서 한국폴리텍대학 인천캠퍼스 가는 지하철 경로 찾아줘'라고 명령하면, 시스템은 브라우저 접속, 검색창 탐색, 입력, 결과 수집, 정보 요약, 후속 질문 생성까지의 과정을 단계적으로 수행한다.

따라서 Voice Navigator는 단순 음성 입력 기능이 아니라, 복합 작업을 계획하고 실행하는 작업형 AI Agent 시스템으로 정의된다.

2. 설계 원칙

시스템은 백그라운드 프로그램과 메인 프로그램으로 구성된 이중 구조를 가져야 한다.
사용자는 단축키, 웨이크 워드, GUI 버튼, 텍스트 입력 등 복수의 입력 경로를 사용할 수 있어야 한다.
사용자의 명령은 바로 실행되는 것이 아니라, 명령 정규화 → 의도 분석 → 슬롯 추출 → 목표 정의 → 단계 계획 생성 → 실행 → 검증 → 결과 요약의 절차를 거쳐 처리되어야 한다.
접근성과 보안을 동시에 고려하여, 민감 입력 환경에서는 음성 안내와 외부 전송을 제한할 수 있어야 한다.

3. 시스템 구조 개요

Voice Navigator는 크게 두 프로그램으로 나뉜다. 백그라운드 프로그램은 사용자의 호출 시도를 감지하고 메인 프로그램을 활성화하는 상시 대기 프로세스이며, 메인 프로그램은 실제 GUI, 설정, STT/TTS, AI Agent 처리, 자동화 엔진 실행, 결과 응답, 로그 관리 기능을 수행하는 주 애플리케이션이다.

5. 백그라운드 프로그램 기능 명세

프로그램명: Voice Navigator Background Service  |  목적: 사용자의 호출 시도를 감지하고, 메인 프로그램의 실행 여부를 확인한 뒤, 필요 시 메인 프로그램을 실행 또는 활성화하는 상시 대기 서비스

기능 ID기능명설명
BG-01메인 프로그램 실행 여부 감지현재 메인 프로그램이 실행 중인지 주기적으로 확인한다. 실행 중이면 포커스 활성화, 미실행이면 실행 준비를 한다.
BG-02글로벌 단축키 감지사용자가 설정한 호출 단축키를 시스템 전역에서 감지한다. 예: F2, Ctrl+Space
BG-03웨이크 워드 감지설정된 호출어를 감지한다. 예: '보이스 내비게이터', '보이스'
BG-04메인 프로그램 활성화호출 감지 시 WAKE_UP, START_LISTENING 이벤트를 메인 프로그램에 전달한다.
BG-05감지 기능 일시 중지메인 프로그램이 Listening 또는 Session Running 상태일 때 트리거 감지를 잠시 중단한다.
BG-06감지 기능 재개메인 프로그램이 작업을 마친 뒤 SESSION_FINISHED를 보내면 감지를 다시 시작한다.
BG-07Windows 시작 시 자동 실행백그라운드 프로그램은 Windows 부팅 시 자동으로 실행되도록 등록된다. 시작 프로그램 폴더 등록 또는 레지스트리 Run 등록 방식을 사용한다.

비기능 요구사항

백그라운드 프로그램은 CPU와 메모리 사용량이 낮아야 하며, 장시간 실행되어도 안정적이어야 한다. 단축키 입력 및 웨이크 워드 감지 후 빠르게 반응해야 하고, 메인 프로그램과의 상태 충돌이 없어야 한다.

사용자 명령: "네이버 지도에서 서울역에서 한국폴리텍대학 인천캠퍼스 가는 지하철 경로 찾아줘"

13. 기술 스택 상세 정리

13.1 GUI — PySide6

PySide6는 메인 프로그램 UI를 담당한다. 상단 상태바, 좌측 기능 패널, 중앙 실행 로그, 하단 텍스트 입력창, 우측 엔진 상태 패널, 설정창, 최근 명령창 등을 구성한다. Python 기반이라 STT, TTS, LLM, Playwright와 동일 언어로 묶기 쉽고, Tkinter보다 구조적인 UI를 만들기에 유리하다.

13.2 웹 자동화 — Playwright

Playwright는 유튜브, 구글, 네이버지도 등 웹 서비스 자동화를 담당한다. 좌표 클릭보다 DOM 기반 자동화가 필수이므로 Playwright가 가장 적합하다. 검색창 탐색, 결과 카드 수집, 경로 결과 확인 등 단계별 검증 로직을 붙이기 쉽다.

13.3 Windows 접근성 — UI Automation

Windows UI Automation은 현재 활성 창, 포커스된 요소, 버튼/입력창 역할, 보안 입력 필드 감지 등을 담당한다. 이 프로젝트가 시각장애인을 위한 PC 음성 비서이기 때문에, 운영체제 차원의 접근성 정보 활용이 필수적이다.

13.4 LLM — GPT-4.1 mini

GPT-4.1 mini는 Planner Agent의 핵심 모델이다. 자연어 명령 정규화, 의도 분석, 검색어/출발지/도착지/교통수단 등의 슬롯 추출, 복합 작업 단계별 계획 변환을 수행한다. 초보 팀 기준으로 복합 명령 품질과 구현 안정성의 균형이 좋다.

13.5 STT — gpt-4o-mini-transcribe + whisper.cpp 병행 전략

STT는 전체 시스템 성공률을 좌우하는 입력 계층이다. 1단계에서는 gpt-4o-mini-transcribe로 전체 파이프라인을 완성하고, 2단계에서는 whisper.cpp를 별도 브랜치로 붙여 한국어 정확도, 반응 속도, CPU 점유율, 실시간성, 배포 편의성을 비교한다. 발표 전까지 whisper.cpp가 안정화되면 전환을 검토하고, 그렇지 않으면 gpt-4o-mini-transcribe를 유지한다.

단계사용 기술목표
1단계gpt-4o-mini-transcribe전체 파이프라인을 빠르게 완성하고 발표 안정성을 확보한다.
2단계whisper.cpp로컬 처리 가능성, 비용 절감 효과, 오프라인 확장 가능성을 검증한다.
3단계최종 선택 판단발표 직전까지 whisper.cpp 안정화 여부를 보고 유지 또는 전환을 결정한다.

13.6 TTS — GCP Text-to-Speech

TTS는 결과를 사용자에게 음성으로 읽어 주는 출력 계층이다. 신규 사용자 무료 크레딧 활용 가능성이 가장 큰 채택 이유이다. 장기 운영을 고려하면 Standard 또는 WaveNet 계열을 우선 사용하는 것이 비용 효율 측면에서 유리하다.

13.7 데이터 구조 — JSON

JSON은 Agent 간 데이터 구조화를 담당한다. intent, platform, slots, goal, task_plan, response_template, execution_result 등을 공통 형식으로 저장하고 주고받아, 시스템 유지보수와 디버깅에 유리하다.

14. AI Agent 내부 JSON 구조 설계

Planner Agent는 사용자 명령을 구조화된 데이터로 바꾸고, Executor와 Verifier가 읽기 쉬운 작업 계획으로 전달한다.

{ "intent": "naver_map_route", "platform": "naver_map", "slots": { "origin": "서울역", "destination": "한국폴리텍대학 인천캠퍼스", "transport": "subway" }, "goal": "네이버지도에서 지하철 경로를 검색하고 결과를 요약하여 사용자에게 안내한다", "task_plan": [ {"step": 1, "action": "open_browser"}, {"step": 2, "action": "search_website", "target": "naver_map"}, {"step": 3, "action": "verify_url", "contains": "map.naver.com"}, {"step": 4, "action": "enter_route_mode"}, {"step": 5, "action": "set_origin", "value": "서울역"}, {"step": 6, "action": "verify_origin", "value": "서울역"}, {"step": 7, "action": "set_destination", "value": "한국폴리텍대학 인천캠퍼스"}, {"step": 8, "action": "verify_destination","value": "한국폴리텍대학 인천캠퍼스"}, {"step": 9, "action": "select_transport", "value": "subway"}, {"step":10, "action": "verify_transport", "value": "subway"}, {"step":11, "action": "extract_route_result"} ], "response_template": { "summary": "{origin}에서 {destination}까지 가는 지하철 경로를 찾았습니다.", "follow_up": "경로를 안내할까요?" } }

Planner는 자연어를 구조화하고, Executor는 task_plan을 순서대로 처리하며, Verifier는 verify 단계에서 상태를 확인한다. Response Agent는 response_template를 기반으로 사용자 응답을 생성한다.

15. Playwright + LLM 실제 코드 예시

import json from openai import OpenAI client = OpenAI() def plan_task(command: str) -> dict: prompt = f""" 사용자 명령을 JSON 작업 계획으로 변환하라. 명령: {command} 반환 형식: {{ "intent": "...", "platform": "...", "slots": {{}}, "task_plan": [] }} """ response = client.responses.create( model="gpt-4.1-mini", input=prompt ) return json.loads(response.output_text) from playwright.sync_api import sync_playwright def execute_youtube_search(keyword: str): with sync_playwright() as p: browser = p.chromium.launch(headless=False) page = browser.new_page() page.goto("https://www.youtube.com") page.fill("input#search", keyword) page.press("input#search", "Enter") page.wait_for_selector("ytd-video-renderer") videos = page.query_selector_all("ytd-video-renderer")[:3] results = [] for video in videos: title_el = video.query_selector("#video-title") time_el = video.query_selector( "badge-shape, span.ytd-thumbnail-overlay-time-status-renderer" ) title = title_el.inner_text().strip() if title_el else "제목 없음" duration = time_el.inner_text().strip() if time_el else "시간 정보 없음" results.append({"title": title, "duration": duration}) return results command = "유튜브 들어가서 고양이가 춤추는 영상 찾아줘" plan = plan_task(command) if plan["intent"] == "youtube_video_search": results = execute_youtube_search(plan["slots"]["keyword"]) for idx, item in enumerate(results, start=1): print(f"{idx}번째 영상: {item['title']} / 재생시간: {item['duration']}")

실제 구현에서는 위 코드에 Verifier Agent가 추가되어 브라우저 실행 여부, URL 정상 접속 여부, 검색 결과 존재 여부 등을 단계별로 확인한다. 또한 19.3절에 따라 Playwright는 별도 스레드의 asyncio 루프에서 실행되며, 위 코드는 구조 이해를 위한 참고 예시이다.

DOM Locator 우선순위 fallback 체인

Playwright는 HTML에 role이 명시되지 않아도 태그의 암묵적 role을 자동으로 인식한다. 단, 대형 사이트는 커스텀 컴포넌트를 사용하는 경우가 많으므로 아래 우선순위 fallback 구조로 요소를 탐색한다.

순위방법사용 상황예시
1순위 get_by_role() button, input 등 표준 HTML 태그. role 미명시 시 암묵적 role 자동 인식 page.get_by_role("textbox", name="출발지")
2순위 get_by_placeholder() placeholder 텍스트가 있는 input page.get_by_placeholder("출발지를 입력하세요")
3순위 get_by_label() label이 연결된 input page.get_by_label("출발지")
4순위 locator() CSS 셀렉터 클래스명, id가 명확한 경우 page.locator("input.origin_input")
5순위 locator() contenteditable div 기반 커스텀 입력 컴포넌트 page.locator("div[contenteditable='true']").first

순서action설명대기 조건

자동완성 후보 선택 전략

사용자의 입력과 완전히 일치하는 자동완성 항목이 없는 경우가 존재한다. 예를 들어 사용자가 "서울역"을 입력하면 자동완성에 "서울역 (공항철도)", "서울역 1호선", "서울역 4호선"과 같이 여러 후보가 나타난다. 이 경우 아래 단계적 전략을 순서대로 적용한다.

순위전략적용 조건예시
1순위 완전 일치 항목 선택 자동완성 목록에 사용자 입력과 완전히 일치하는 항목이 있을 때 "서울역" → "서울역" 항목 바로 선택
2순위 슬롯 기반 규칙 매칭 완전 일치 없음. slots의 transport 등 정보로 후보를 필터링할 수 있을 때 transport=subway → "서울역 1호선" 우선 선택
3순위 LLM 판단 규칙 매칭도 불명확할 때. 후보 목록 전체를 LLM에 전달하여 사용자 의도에 가장 맞는 항목 선택 후보 목록 + 사용자 명령 + slots → LLM이 최적 항목 반환
4순위 사용자 선택 요청 LLM도 판단 불가할 때. 후보 목록을 TTS로 읽어주고 사용자가 직접 선택 "서울역 검색 결과가 여러 개예요. 1번 공항철도, 2번 1호선, 3번 4호선. 몇 번으로 할까요?"
자동완성 미등장 시 fallback: wait_for_autocomplete가 timeout되면 자동완성 없이 Enter 직접 입력을 시도한다. Enter 후에도 결과가 없으면 execution_result에 type: "autocomplete_not_found"로 기록하고 재시도 정책을 적용한다.

재시도 정책

재시도는 스텝 단위로 수행한다. 실패한 검증 스텝 전체를 처음부터 반복하는 것이 아니라, 해당 검증과 연관된 선행 스텝(retry_from)부터 재실행한다. 각 스텝의 최대 재시도 횟수는 3회로 제한한다.

항목내용
재시도 단위스텝 단위 (전체 작업 재시도 아님)
재시도 시작 지점각 verify 스텝의 retry_from 필드에 정의된 스텝부터 재실행
최대 재시도 횟수3회 (MAX_RETRY = 3)
3회 전부 실패 시execution_result 생성 → Response Agent에 전달 → TTS로 사용자 안내 → fallback_action 실행 또는 종료
무한루프 방지retry.count가 MAX_RETRY 도달 시 즉시 실패 처리. 재시도 중 추가 재시도 불허

execution_result JSON 스키마

Verifier Agent는 검증 결과를 아래 구조로 생성하며, Response Agent는 이를 기반으로 사용자 응답 및 TTS 안내를 생성한다.

{ "execution_result": { "status": "success | failed | retrying", "verification_source": "dom | uiautomation | filesystem", "failed_step": null, "failed_action": null, "error": { "type": "timeout | value_mismatch | element_not_found | autocomplete_not_found | page_load_failed | process_not_found | file_not_found | null", "reason": null, "actual_value": null }, "retry": { "count": 0, "max": 3, "retried_from_step": null }, "response_hint": { "fallback_action": "suggest_keyword_change | confirm_with_user | press_enter_fallback | null", "user_message": null } } }

실패 유형별 response_hint 적용 기준은 아래와 같다.

error.typeuser_message 방향fallback_action
timeout 네트워크 또는 사이트 응답 문제 안내 press_enter_fallback
value_mismatch actual_value를 읽어 사용자에게 현재 입력값 확인 요청 confirm_with_user
element_not_found 사이트 구조 변경 가능성 안내 후 재시도 null (재시도 후 종료)

전체 흐름 요약: Executor 실행 → Verifier 검증 → 성공: execution_result(status: success) → 다음 스텝 진행 → 실패: retry_from 스텝부터 재시도 (최대 3회) → 3회 전부 실패: execution_result(status: failed) 생성 → error.type / actual_value / response_hint 채움 → Response Agent가 user_message를 TTS로 출력 → fallback_action 실행 또는 작업 종료

19.3 PySide6 + Playwright 비동기 구조

PySide6는 Qt 이벤트 루프를, Playwright 비동기 버전은 asyncio 이벤트 루프를 사용한다. 두 루프는 하나의 스레드에서 동시에 실행될 수 없으므로 반드시 스레드를 분리해야 한다. 이 구조는 개발 1단계에서 골격을 확정하며, 나중에 변경하면 전체 구조를 재작성해야 하므로 초반에 반드시 설계해둔다.

역할담당 스레드통신 방법
Qt 이벤트 루프 (GUI 관리)메인 스레드작업 요청: queue.Queue / 결과 반환: PySide6 Signal
asyncio 이벤트 루프 (Playwright)별도 스레드 (QThread 또는 threading.Thread)
Qt 이벤트 루프: Voice Navigator 창 안의 버튼 클릭, 텍스트 입력, 스크롤 등 메인 프로그램 창 내부 이벤트만 감지한다. 외부 브라우저나 다른 프로그램과는 무관하다. 이 루프가 멈추면 창이 응답 없음 상태가 된다.

asyncio 이벤트 루프: Playwright의 브라우저 조작 중 발생하는 대기 구간(페이지 로딩, 요소 등장 등)을 효율적으로 관리한다. 메인 스레드와 독립된 스레드에서 실행되어 GUI가 굳지 않는다.

방법문제점판단

22.2 보안 입력 모드 — 비밀번호 필드 감지 시 동작

Windows UI Automation을 통해 현재 포커스된 입력창이 비밀번호 필드인지 감지한다. 감지 즉시 보안 입력 모드를 활성화하며, 아래 동작을 수행한다.

순서동작이유

23. 화면 읽기 기능 상세 정의

Voice Navigator의 화면 읽기 기능은 텍스트 기반으로만 동작한다. 이미지 설명 기능은 본 프로젝트 범위에서 배제한다. 웹 페이지, Windows 기본 앱, 써드파티 앱(카카오톡 등) 화면의 텍스트를 추출하여 TTS로 안내하는 것을 목표로 한다.

23.2 웹 페이지 본문 추출 전략

웹 페이지마다 본문 구조가 달라 단일 셀렉터로 처리할 수 없다. 아래 2단계 전략을 순서대로 적용한다.

단계방식동작적용 조건
1단계 규칙 기반 추출 article, main, [role=main], .content 등 본문 관련 셀렉터 순서대로 시도. header, footer, nav, script, style 등 제거 항상 먼저 시도
2단계 LLM 판단 fallback 규칙 기반 추출 결과가 비어있거나 너무 짧을 때 페이지 전체 텍스트를 LLM에 전달하여 주요 내용 추출 요청 1단계 결과 불충분 시
본문 추출 우선순위 셀렉터: article → main → [role="main"] → .content → .post-body → #content → body (최후 수단)
제거 대상: header, footer, nav, aside, .advertisement, .sidebar, script, style

Reading 상태에서는 아래 키워드를 STT 변환 직후 Planner 호출 없이 즉시 처리한다. 일반 명령 처리 파이프라인을 우회하여 응답 지연을 최소화한다.

키워드 예시intent동작상태 변화

23.5 섹션 단위 읽기 구조

"다음"과 "처음부터" 명령을 지원하려면 추출된 텍스트를 섹션 단위로 분할하여 인덱스로 관리해야 한다. 섹션은 h1~h3, p 태그 단위로 분할하며, 현재 섹션 인덱스를 메인 프로그램 내부 상태로 유지한다.

항목내용
섹션 분할 기준h1, h2, h3, p 태그 단위로 분할. 빈 텍스트 제외
섹션 인덱스 관리current_section 변수로 현재 위치 추적
"다음" 처리current_section += 1 → 해당 섹션 TTS 출력
"처음부터" 처리current_section = 0 → 첫 섹션부터 TTS 출력
마지막 섹션 완료 시TTS: "읽기가 완료됐어요." → Ready 상태 복귀

웹이 아닌 네이티브 앱의 화면 읽기는 UI Automation을 통해 현재 활성 창의 텍스트 요소를 탐색하여 추출한다.

대상추출 방식추출 요소fallback
